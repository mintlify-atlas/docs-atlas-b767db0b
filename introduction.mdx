HTTPower is a production-ready HTTP client library for Elixir. It sits above your existing HTTP client — Finch, Req, or Tesla — and adds the reliability patterns you need in production: circuit breakers, rate limiting, PCI-compliant logging, smart retries, request deduplication, and comprehensive telemetry.

## Why HTTPower?

Elixir's HTTP ecosystem has excellent clients, but each solves a different problem:

- **Finch** gives you high-throughput connection pooling built on Mint
- **Req** gives you batteries-included convenience with auto-JSON and compression
- **Tesla** gives you a composable middleware pipeline

None of them give you production reliability out of the box. When you call a payment gateway, a third-party API, or a downstream microservice, you need circuit breakers to stop cascading failures, rate limiting to respect API quotas, retries with exponential backoff, and logs that don't leak card numbers.

HTTPower adds all of that as a consistent layer above whichever HTTP client you already use.

## When to use HTTPower

**Payment processing** — PCI-compliant logging sanitizes card numbers, CVV codes, and authorization headers automatically. Circuit breakers prevent your checkout flow from hanging when a payment gateway degrades. Deduplication stops double-charges from impatient users.

**High-volume API integrations** — Rate limiting with token bucket keeps you within API quotas. Smart retries with jitter spread load during 429 bursts. Telemetry gives you per-endpoint request rates and durations.

**Microservices** — Per-service circuit breakers with isolated keys prevent one failing service from taking down others. Correlation IDs trace requests across service boundaries in distributed logs.

## Architecture overview

HTTPower has two layers:

**Adapter layer** — A thin behaviour (`HTTPower.Adapter`) that normalizes request and response shapes across Finch, Req, and Tesla. Each adapter translates HTTPower's standardized options to the underlying client's format.

**Middleware pipeline** — HTTPower's reliability features run above the adapter. When you call `HTTPower.get/2`, the request passes through rate limiting, circuit breaker, and deduplication checks before reaching the adapter. Retry logic wraps the adapter call and re-executes on failure.

```
HTTPower.get(url, opts)
        │
        ▼
┌─────────────────────┐
│  Rate limiter       │  Token bucket, wait/error strategies
├─────────────────────┤
│  Circuit breaker    │  Closed → Open → Half-open state machine
├─────────────────────┤
│  Deduplication      │  In-flight fingerprint tracking
├─────────────────────┤
│  Retry logic        │  Exponential backoff with jitter
├─────────────────────┤
│  Adapter            │  Finch / Req / Tesla
└─────────────────────┘
        │
        ▼
{:ok, %HTTPower.Response{}} | {:error, %HTTPower.Error{}}
```

Configuration flows from global application config → per-client options → per-request options, with more specific values always taking precedence.
