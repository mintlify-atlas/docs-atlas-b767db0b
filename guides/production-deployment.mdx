This guide covers everything you need to run HTTPower reliably in production: configuration, supervision, security, performance tuning, and graceful degradation.

## Supervision tree

HTTPower starts its own supervision tree automatically when you add it as a dependency. You don't need to add any children to your own supervisor.

The tree manages two GenServers that use ETS for fast in-memory state:

- `HTTPower.Middleware.RateLimiter` — token bucket state
- `HTTPower.Middleware.CircuitBreaker` — circuit state per service key

```
YourApp.Supervisor
├── HTTPower.Supervisor (one_for_one)
│   ├── HTTPower.Middleware.RateLimiter
│   └── HTTPower.Middleware.CircuitBreaker
└── ... your other children
```

The `one_for_one` restart strategy means a crash in one GenServer restarts only that process — not the entire HTTPower tree.

### Verifying the supervision tree

Confirm HTTPower's processes are running in IEx:

```elixir
iex> Process.whereis(HTTPower.Middleware.RateLimiter)
#PID<0.234.0>

iex> Process.whereis(HTTPower.Middleware.CircuitBreaker)
#PID<0.235.0>

iex> Supervisor.which_children(HTTPower.Supervisor)
[
  {HTTPower.Middleware.CircuitBreaker, #PID<0.235.0>, :worker, [HTTPower.Middleware.CircuitBreaker]},
  {HTTPower.Middleware.RateLimiter, #PID<0.234.0>, :worker, [HTTPower.Middleware.RateLimiter]}
]
```

## Profiles

HTTPower ships with three pre-configured profiles that encode best practices for common scenarios. Use a profile as your starting point and override individual settings as needed.

```elixir
client = HTTPower.new(
  base_url: "https://api.example.com",
  profile: :payment_processing
)

# Explicit options override the profile
client = HTTPower.new(
  base_url: "https://api.example.com",
  profile: :high_volume_api,
  rate_limit: [requests: 2000]  # overrides profile's 1000 req/min
)
```

<AccordionGroup>
  <Accordion title=":payment_processing">
    Optimized for Stripe, PayPal, and other payment gateways where correctness matters more than speed.

    | Setting | Value | Rationale |
    |---|---|---|
    | `rate_limit.requests` | 100/min | Conservative |
    | `rate_limit.adaptive` | `true` | Reduces rate when circuit is struggling |
    | `circuit_breaker.failure_threshold_percentage` | 30% | Aggressive — payment APIs must be reliable |
    | `circuit_breaker.timeout` | 30s | Payment APIs can be slow to recover |
    | `deduplicate.ttl` | 5s | Prevents double charges |
    | `max_retries` | 3 | With 2s base delay |

    ```elixir
    client = HTTPower.new(
      base_url: "https://api.stripe.com",
      profile: :payment_processing
    )
    ```
  </Accordion>

  <Accordion title=":high_volume_api">
    Optimized for public APIs or high-traffic internal services where throughput is the priority.

    | Setting | Value | Rationale |
    |---|---|---|
    | `rate_limit.requests` | 1000/min | High throughput |
    | `circuit_breaker.failure_threshold_percentage` | 50% | Tolerant — expect some failures at volume |
    | `circuit_breaker.timeout` | 5s | Fast recovery |
    | `circuit_breaker.half_open_requests` | 10 | Batch test requests |
    | `deduplicate.ttl` | 1s | Short window, response sharing |
    | `max_retries` | 2 | Fast delays (500ms base) |

    ```elixir
    client = HTTPower.new(
      base_url: "https://content-api.example.com",
      profile: :high_volume_api
    )
    ```
  </Accordion>

  <Accordion title=":microservices_mesh">
    Optimized for service-to-service calls in Kubernetes or a service mesh, where preventing cascade failures is the top concern.

    | Setting | Value | Rationale |
    |---|---|---|
    | `rate_limit.requests` | 500/min | Moderate with adaptive |
    | `circuit_breaker.failure_threshold_percentage` | 40% | Balanced |
    | `circuit_breaker.timeout` | 10s | Medium recovery window |
    | `deduplicate.ttl` | 2s | Critical for retry storms |
    | `max_retries` | 3 | Standard with 1s base delay |

    ```elixir
    defmodule MyApp.UserService do
      def client do
        HTTPower.new(
          base_url: "https://user-service.internal",
          profile: :microservices_mesh,
          circuit_breaker_key: "user_service"
        )
      end
    end
    ```
  </Accordion>
</AccordionGroup>

## Recommended production configuration

Add this to `config/prod.exs` as your production baseline:

```elixir
# config/prod.exs
config :httpower,
  # Retry
  max_retries: 3,
  retry_safe: false,
  base_delay: 1000,
  max_delay: 30_000,

  # Circuit breaker
  circuit_breaker: [
    enabled: true,
    failure_threshold: 5,
    window_size: 10,
    timeout: 60_000,
    half_open_requests: 1
  ],

  # Rate limiting
  rate_limit: [
    enabled: true,
    requests: 100,
    per: :minute,
    strategy: :wait,
    max_wait_time: 5000
  ],

  # Logging (attach separately in application.ex)
  logging: [
    level: :info,
    sanitize_headers: ["authorization", "api-key", "x-api-key", "cookie"],
    sanitize_body_fields: [
      "password", "secret", "token",
      "credit_card", "card_number", "cvv", "cvc",
      "ssn", "tax_id"
    ]
  ]
```

### Per-service configuration

For applications that call multiple external services, configure each client independently:

```elixir
# config/prod.exs
config :myapp, :stripe_client,
  circuit_breaker: [failure_threshold: 3, timeout: 30_000],
  rate_limit: [requests: 100, per: :second]

config :myapp, :github_client,
  circuit_breaker: [failure_threshold: 5, timeout: 60_000],
  rate_limit: [requests: 60, per: :minute]

# lib/my_app/stripe_client.ex
defmodule MyApp.StripeClient do
  def client do
    config = Application.get_env(:myapp, :stripe_client, [])

    HTTPower.new([
      base_url: "https://api.stripe.com",
      circuit_breaker_key: "stripe"
    ] ++ config)
  end
end
```

## Environment variables

Use `config/runtime.exs` for secrets and environment-specific values:

```elixir
# config/runtime.exs
import Config

if config_env() == :prod do
  config :myapp, :api_client,
    base_url: System.get_env("API_BASE_URL") || "https://api.example.com",
    api_key: System.fetch_env!("API_KEY"),
    timeout: String.to_integer(System.get_env("API_TIMEOUT", "30")),
    max_retries: String.to_integer(System.get_env("API_MAX_RETRIES", "3"))
end
```

<Warning>
Never put API keys or secrets directly in config files. Use `System.fetch_env!/1` in `runtime.exs` so secrets are read at startup, not at compile time.
</Warning>

## Logging setup

Attach the built-in PCI-compliant logger in your `Application.start/2` callback:

```elixir
# lib/my_app/application.ex
defmodule MyApp.Application do
  use Application

  def start(_type, _args) do
    HTTPower.Logger.attach(
      level: :info,
      log_headers: true,
      log_body: true
    )

    children = [
      # ... your supervision tree
    ]

    Supervisor.start_link(children, strategy: :one_for_one, name: MyApp.Supervisor)
  end
end
```

The logger automatically:
- Sanitizes `Authorization`, `Cookie`, `API-Key`, and similar headers
- Redacts credit card numbers (Luhn-validated), CVV codes, passwords, and SSNs
- Sets structured `Logger.metadata/0` on every log entry for log aggregation
- Generates correlation IDs in the format `req_<16 hex chars>` (or reuses Phoenix's `request_id`)

### Structured metadata queries

All HTTPower log entries include machine-readable metadata. Use these in Datadog, Splunk, ELK, or Loki:

```
# Slow requests
httpower_duration_ms:>1000 AND httpower_event:response

# 5xx errors on POST endpoints
httpower_status:>=500 AND httpower_method:post

# Trace a single request end-to-end
httpower_correlation_id:"req_abc123"
```

Available fields: `httpower_correlation_id`, `httpower_event`, `httpower_method`, `httpower_url`, `httpower_status`, `httpower_duration_ms`, `httpower_headers`, `httpower_response_headers`, `httpower_body`, `httpower_response_body`.

## SSL and security

### SSL verification

SSL verification is enabled by default. Never disable it in production:

```elixir
# Default — no action needed
config :httpower, ssl_verify: true
```

### Sensitive data logging

Configure comprehensive sanitization for PCI compliance:

```elixir
config :httpower,
  logging: [
    level: :info,
    log_headers: true,
    log_body: true,
    sanitize_headers: [
      "authorization", "api-key", "x-api-key",
      "cookie", "set-cookie", "x-auth-token"
    ],
    sanitize_body_fields: [
      # Authentication
      "password", "secret", "token", "api_key",
      # Payment
      "credit_card", "card_number", "cvv", "cvc",
      "account_number", "routing_number",
      # Personal
      "ssn", "tax_id", "drivers_license"
    ]
  ]
```

### Test mode

Never enable test mode in production:

```elixir
# config/prod.exs
config :httpower, test_mode: false  # default — add this explicitly for clarity

# config/test.exs only
config :httpower, test_mode: true
```

## Timeout tuning

Match your timeout to the API's expected response time:

```elixir
# Fast API (< 10s)
client = HTTPower.new(base_url: "https://fast-api.example.com", timeout: 10)

# Slow API or large payloads
client = HTTPower.new(base_url: "https://slow-api.example.com", timeout: 120)

# Override per request for known slow endpoints
HTTPower.get(client, "/export", timeout: 300)
```

### Connection pooling

Connection pooling is managed by the underlying adapter.

<Tabs>
  <Tab title="Finch (default)">
    Finch manages connection pools automatically. Configure pool size for high-throughput workloads:

    ```elixir
    # config/config.exs
    config :my_app, MyApp.Finch,
      pools: %{
        "https://api.example.com" => [size: 25, count: 5],
        :default => [size: 10, count: 1]
      }
    ```
  </Tab>
  <Tab title="Req">
    Req delegates to Finch internally. Configure Finch pools as above.

    ```elixir
    config :req, finch: [
      pools: %{
        default: [size: 25, count: 5]
      }
    ]
    ```
  </Tab>
  <Tab title="Tesla">
    Configure the pool on your Tesla adapter (Hackney, Finch, etc.).
  </Tab>
</Tabs>

## Multi-node considerations

HTTPower's rate limiter and circuit breaker state is stored in local ETS — each node has independent state.

**Rate limiting:**
- Total throughput = `requests_per_node × num_nodes`
- Example: 100 req/min × 3 nodes = 300 req/min globally

**Circuit breaker:**
- Each node opens independently — this is usually the desired behavior
- One node experiencing failures doesn't affect other nodes

For strict global rate limits across nodes, implement centralized rate limiting with Redis or a distributed state library alongside HTTPower's per-node limiter.

### Load balancer configuration

```elixir
config :httpower,
  timeout: 30,
  circuit_breaker: [enabled: true],
  rate_limit: [
    enabled: true,
    requests: 50  # Divide your global limit by node count
  ]
```

## Monitoring

### Circuit breaker state

Monitor circuit state changes in production:

```elixir
defmodule MyApp.CircuitBreakerMonitor do
  require Logger

  def check_circuits do
    circuits = ["payment_api", "user_service", "order_service"]

    Enum.each(circuits, fn circuit_key ->
      case HTTPower.Middleware.CircuitBreaker.get_state(circuit_key) do
        :open -> Logger.warning("Circuit OPEN: #{circuit_key}")
        :half_open -> Logger.info("Circuit HALF-OPEN: #{circuit_key}")
        :closed -> :ok
        nil -> :ok
      end
    end)
  end
end
```

Schedule this with Quantum, Oban, or a periodic GenServer to get alertable metrics on circuit state.

### ETS memory usage

HTTPower's ETS tables are self-maintaining:
- Rate limiter cleans up buckets inactive for 5 minutes
- Circuit breaker tracks only the last `window_size` requests
- Typical memory usage is under 10 MB

If you want to inspect table sizes:

```elixir
:ets.info(:httpower_rate_limiter)
:ets.info(:httpower_circuit_breaker)
```

## Graceful degradation

Handle circuit breaker and rate limit errors explicitly:

```elixir
defmodule MyApp.PaymentService do
  def charge_customer(amount) do
    case HTTPower.post(payment_client(), "/charge", body: amount) do
      {:ok, response} ->
        {:ok, response}

      {:error, %{reason: :service_unavailable}} ->
        Logger.error("Payment service unavailable (circuit open)")
        {:error, :service_unavailable}

      {:error, %{reason: :too_many_requests}} ->
        Logger.warning("Payment service rate limited")
        {:error, :rate_limited}

      {:error, error} ->
        {:error, error}
    end
  end

  defp payment_client do
    HTTPower.new(
      base_url: "https://api.stripe.com",
      profile: :payment_processing
    )
  end
end
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="Circuit keeps opening">
    **Debug:**

    ```elixir
    HTTPower.Middleware.CircuitBreaker.get_state("my_service")
    # Look in logs for: [HTTPower] [req_*] ← 500
    ```

    **Solutions:**
    - Increase `failure_threshold` if the service has transient issues
    - Increase `window_size` to smooth out spikes
    - Reduce `timeout` if the service is responding slowly
    - Verify the external service is actually reachable
  </Accordion>

  <Accordion title="Rate limiting too aggressive">
    **Debug:**

    ```elixir
    Application.get_env(:httpower, :rate_limit)
    ```

    **Solutions:**
    - Increase `requests` or change `per` to a larger window
    - Switch to `:error` strategy and handle `{:error, :too_many_requests}` in your code
    - Use `rate_limit_key` to give different endpoints separate buckets
  </Accordion>

  <Accordion title="High memory usage">
    Inspect the ETS tables:

    ```elixir
    :ets.info(:httpower_rate_limiter) |> Keyword.get(:size)
    :ets.info(:httpower_circuit_breaker) |> Keyword.get(:size)
    ```

    The rate limiter runs automatic cleanup every minute and removes buckets inactive for 5 minutes. If memory is still growing, check for a large number of unique `rate_limit_key` values.
  </Accordion>
</AccordionGroup>

## Deployment checklist

<Steps>
  <Step title="Configure circuit breaker thresholds">
    Set `failure_threshold` and `timeout` based on each service's SLA and typical recovery time.
  </Step>
  <Step title="Set per-service rate limits">
    Configure rate limits to stay within each external API's documented limits.
  </Step>
  <Step title="Configure PCI-compliant logging">
    Attach `HTTPower.Logger` with appropriate `sanitize_body_fields` for any sensitive data your services exchange.
  </Step>
  <Step title="Disable test mode">
    Confirm `config :httpower, test_mode: false` in `prod.exs`.
  </Step>
  <Step title="Verify SSL">
    Confirm `ssl_verify: true` (the default) is not overridden anywhere in production config.
  </Step>
  <Step title="Set environment variables">
    Move all API keys to `System.fetch_env!/1` in `runtime.exs`.
  </Step>
  <Step title="Test failure scenarios">
    Simulate circuit breaker opens and rate limit hits in staging before deploying.
  </Step>
  <Step title="Set up monitoring">
    Schedule circuit breaker state checks and alert on `:open` state changes.
  </Step>
</Steps>
