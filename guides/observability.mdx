HTTPower emits telemetry events for every HTTP request, retry, rate limit check, circuit breaker transition, and deduplication decision using Elixir's built-in `:telemetry` library. This guide covers all events and how to integrate them with your observability stack.

## Quick start

Attach a handler to start receiving events:

```elixir
:telemetry.attach_many(
  "httpower-handler",
  [
    [:httpower, :request, :start],
    [:httpower, :request, :stop],
    [:httpower, :retry, :attempt]
  ],
  fn event, measurements, metadata, _config ->
    IO.inspect({event, measurements, metadata})
  end,
  nil
)
```

## Event reference

### HTTP request lifecycle

HTTPower uses `:telemetry.span/3` to emit start, stop, and exception events around every HTTP request.

<AccordionGroup>
  <Accordion title="[:httpower, :request, :start]">
    Emitted when a request begins.

    **Measurements:**

    | Key | Type | Description |
    |---|---|---|
    | `system_time` | integer | System time at request start |

    **Metadata:**

    | Key | Type | Description |
    |---|---|---|
    | `method` | atom | HTTP method — `:get`, `:post`, `:put`, `:delete` |
    | `url` | string | Sanitized URL (query params and fragments stripped) |

    **Example:**

    ```elixir
    event: [:httpower, :request, :start]
    measurements: %{system_time: 1640995200000000000}
    metadata: %{method: :get, url: "https://api.example.com/users"}
    ```
  </Accordion>

  <Accordion title="[:httpower, :request, :stop]">
    Emitted when a request completes, whether successful or not.

    **Measurements:**

    | Key | Type | Description |
    |---|---|---|
    | `duration` | integer | Duration in native time units — convert with `System.convert_time_unit/3` |
    | `monotonic_time` | integer | Monotonic time at completion |

    **Metadata:**

    | Key | Type | Description |
    |---|---|---|
    | `method` | atom | HTTP method |
    | `url` | string | Sanitized URL |
    | `status` | integer | HTTP status code (success responses) |
    | `error_type` | atom | Error reason (failed requests) |
    | `retry_count` | integer | Number of retries performed (default `0`) |

    **Example (success):**

    ```elixir
    event: [:httpower, :request, :stop]
    measurements: %{duration: 45_000_000, monotonic_time: -576460751234567890}
    metadata: %{
      method: :get,
      url: "https://api.example.com/users",
      status: 200,
      retry_count: 0
    }
    ```

    **Example (error):**

    ```elixir
    event: [:httpower, :request, :stop]
    measurements: %{duration: 5_000_000, monotonic_time: -576460751234567890}
    metadata: %{
      method: :post,
      url: "https://api.example.com/orders",
      error_type: :timeout,
      retry_count: 3
    }
    ```
  </Accordion>

  <Accordion title="[:httpower, :request, :exception]">
    Emitted when an unhandled exception occurs during request processing. HTTPower catches most exceptions and converts them to `{:error, reason}` tuples, so this event is rare.

    **Measurements:**

    | Key | Type | Description |
    |---|---|---|
    | `duration` | integer | Duration in native time units |

    **Metadata:**

    | Key | Type | Description |
    |---|---|---|
    | `method` | atom | HTTP method |
    | `url` | string | Sanitized URL |
    | `kind` | atom | Exception kind — `:throw`, `:error`, `:exit` |
    | `reason` | term | Exception reason |
    | `stacktrace` | list | Exception stacktrace |
  </Accordion>
</AccordionGroup>

### Retry events

<AccordionGroup>
  <Accordion title="[:httpower, :retry, :attempt]">
    Emitted before each retry attempt.

    **Measurements:**

    | Key | Type | Description |
    |---|---|---|
    | `attempt_number` | integer | Current attempt — `2` for first retry, `3` for second |
    | `delay_ms` | integer | Delay before this retry in milliseconds |

    **Metadata:**

    | Key | Type | Description |
    |---|---|---|
    | `method` | atom | HTTP method |
    | `url` | string | Sanitized URL |
    | `reason` | term | Reason for retry — `:timeout`, `{:http_status, 503}`, etc. |

    **Example:**

    ```elixir
    event: [:httpower, :retry, :attempt]
    measurements: %{attempt_number: 2, delay_ms: 1000}
    metadata: %{
      method: :get,
      url: "https://api.example.com/users",
      reason: {:http_status, 503}
    }
    ```
  </Accordion>
</AccordionGroup>

### Rate limiter events

<AccordionGroup>
  <Accordion title="[:httpower, :rate_limit, :ok]">
    Emitted when a request passes the rate limit check.

    **Measurements:**

    | Key | Type | Description |
    |---|---|---|
    | `tokens_remaining` | float | Tokens left in the bucket |
    | `wait_time_ms` | integer | Always `0` for this event |

    **Metadata:**

    | Key | Type | Description |
    |---|---|---|
    | `bucket_key` | string | Rate limit bucket identifier |

    **Example:**

    ```elixir
    event: [:httpower, :rate_limit, :ok]
    measurements: %{tokens_remaining: 45.5, wait_time_ms: 0}
    metadata: %{bucket_key: "api.example.com"}
    ```
  </Accordion>

  <Accordion title="[:httpower, :rate_limit, :wait]">
    Emitted when the rate limit is exceeded and the `:wait` strategy causes a delay.

    **Measurements:**

    | Key | Type | Description |
    |---|---|---|
    | `wait_time_ms` | integer | Time spent waiting for tokens |

    **Metadata:**

    | Key | Type | Description |
    |---|---|---|
    | `bucket_key` | string | Rate limit bucket identifier |
    | `strategy` | atom | Always `:wait` for this event |

    **Example:**

    ```elixir
    event: [:httpower, :rate_limit, :wait]
    measurements: %{wait_time_ms: 250}
    metadata: %{bucket_key: "api.example.com", strategy: :wait}
    ```
  </Accordion>

  <Accordion title="[:httpower, :rate_limit, :exceeded]">
    Emitted when the rate limit is exceeded and the `:error` strategy rejects the request.

    **Measurements:**

    | Key | Type | Description |
    |---|---|---|
    | `tokens_remaining` | integer | Always `0` |

    **Metadata:**

    | Key | Type | Description |
    |---|---|---|
    | `bucket_key` | string | Rate limit bucket identifier |
    | `strategy` | atom | Always `:error` for this event |

    **Example:**

    ```elixir
    event: [:httpower, :rate_limit, :exceeded]
    measurements: %{tokens_remaining: 0}
    metadata: %{bucket_key: "api.example.com", strategy: :error}
    ```
  </Accordion>
</AccordionGroup>

### Circuit breaker events

<AccordionGroup>
  <Accordion title="[:httpower, :circuit_breaker, :state_change]">
    Emitted when the circuit transitions between states.

    **Measurements:**

    | Key | Type | Description |
    |---|---|---|
    | `timestamp` | integer | System timestamp of the transition |

    **Metadata:**

    | Key | Type | Description |
    |---|---|---|
    | `circuit_key` | string | Circuit identifier |
    | `from_state` | atom | Previous state — `:closed`, `:open`, `:half_open` |
    | `to_state` | atom | New state |
    | `failure_count` | integer | Failures in the current window |
    | `failure_rate` | float | Failure rate as decimal, e.g., `0.6` = 60% |

    **Example:**

    ```elixir
    event: [:httpower, :circuit_breaker, :state_change]
    measurements: %{timestamp: 1640995200000000000}
    metadata: %{
      circuit_key: "payment.api.example.com",
      from_state: :closed,
      to_state: :open,
      failure_count: 5,
      failure_rate: 0.83
    }
    ```
  </Accordion>

  <Accordion title="[:httpower, :circuit_breaker, :open]">
    Emitted when a request is blocked because the circuit is open.

    **Measurements:** empty map

    **Metadata:**

    | Key | Type | Description |
    |---|---|---|
    | `circuit_key` | string | Circuit identifier |

    **Example:**

    ```elixir
    event: [:httpower, :circuit_breaker, :open]
    measurements: %{}
    metadata: %{circuit_key: "payment.api.example.com"}
    ```
  </Accordion>
</AccordionGroup>

### Deduplication events

<AccordionGroup>
  <Accordion title="[:httpower, :dedup, :execute]">
    Emitted when a request is the first occurrence and will be executed normally.

    **Measurements:** empty map

    **Metadata:**

    | Key | Type | Description |
    |---|---|---|
    | `dedup_key` | string | SHA-256 hash of `method + URL + body` |
  </Accordion>

  <Accordion title="[:httpower, :dedup, :wait]">
    Emitted when a duplicate request waits for an in-flight request to complete.

    **Measurements:**

    | Key | Type | Description |
    |---|---|---|
    | `wait_time_ms` | integer | Time spent waiting in milliseconds |

    **Metadata:**

    | Key | Type | Description |
    |---|---|---|
    | `dedup_key` | string | Deduplication hash |
  </Accordion>

  <Accordion title="[:httpower, :dedup, :cache_hit]">
    Emitted when a request returns a cached response within the TTL window.

    **Measurements:** empty map

    **Metadata:**

    | Key | Type | Description |
    |---|---|---|
    | `dedup_key` | string | Deduplication hash |
  </Accordion>
</AccordionGroup>

## Setting up telemetry handlers

### Built-in PCI-compliant logger

For structured request/response logging, attach HTTPower's built-in logger in your `Application.start/2`:

```elixir
# lib/my_app/application.ex
def start(_type, _args) do
  HTTPower.Logger.attach(
    level: :info,
    log_headers: true,
    log_body: true
  )

  # ...
end
```

Log output:

```
[HTTPower] [req_a1b2c3d4] → GET https://api.example.com/users headers=%{"authorization" => "[REDACTED]"}
[HTTPower] [req_a1b2c3d4] ← 200 (45ms)
```

### Custom telemetry handler

For custom logging or metrics collection:

```elixir
defmodule MyApp.HTTPowerLogger do
  require Logger

  def setup do
    :telemetry.attach_many(
      "httpower-logger",
      [[:httpower, :request, :stop]],
      &handle_event/4,
      nil
    )
  end

  def handle_event([:httpower, :request, :stop], measurements, metadata, _config) do
    duration_ms = System.convert_time_unit(measurements.duration, :native, :millisecond)
    status = Map.get(metadata, :status, "error")
    method = metadata.method |> to_string() |> String.upcase()

    Logger.info("HTTP #{method} #{metadata.url} - #{status} (#{duration_ms}ms)")
  rescue
    exception ->
      Logger.error("Telemetry handler error: #{inspect(exception)}")
  end
end

# In application.ex
MyApp.HTTPowerLogger.setup()
```

<Note>
Always add a `rescue` block in telemetry handlers. A crashing handler will be detached by `:telemetry` and stop receiving events silently.
</Note>

## Prometheus integration

Export HTTPower metrics to Prometheus using `telemetry_metrics` and `telemetry_metrics_prometheus`:

```elixir
# mix.exs
defp deps do
  [
    {:httpower, "~> 0.9"},
    {:telemetry_metrics, "~> 1.0"},
    {:telemetry_metrics_prometheus, "~> 1.1"}
  ]
end
```

```elixir
# lib/my_app/telemetry.ex
defmodule MyApp.Telemetry do
  use Supervisor
  import Telemetry.Metrics

  def start_link(arg) do
    Supervisor.start_link(__MODULE__, arg, name: __MODULE__)
  end

  def init(_arg) do
    children = [
      {TelemetryMetricsPrometheus, [metrics: metrics()]}
    ]

    Supervisor.init(children, strategy: :one_for_one)
  end

  defp metrics do
    [
      # Request duration histogram
      distribution(
        "httpower.request.duration",
        event_name: [:httpower, :request, :stop],
        measurement: :duration,
        unit: {:native, :millisecond},
        tags: [:method, :status],
        tag_values: fn metadata ->
          %{
            method: metadata.method,
            status: Map.get(metadata, :status, "error")
          }
        end
      ),

      # Request count
      counter(
        "httpower.request.count",
        event_name: [:httpower, :request, :stop],
        tags: [:method, :status]
      ),

      # Retry attempts
      counter(
        "httpower.retry.attempts",
        event_name: [:httpower, :retry, :attempt],
        tags: [:method, :reason],
        tag_values: fn metadata ->
          %{
            method: metadata.method,
            reason: extract_reason(metadata.reason)
          }
        end
      ),

      # Circuit breaker state changes
      counter(
        "httpower.circuit_breaker.state_changes",
        event_name: [:httpower, :circuit_breaker, :state_change],
        tags: [:from_state, :to_state]
      ),

      # Rate limit exceeded
      counter(
        "httpower.rate_limit.exceeded",
        event_name: [:httpower, :rate_limit, :exceeded],
        tags: [:bucket_key]
      )
    ]
  end

  defp extract_reason({:http_status, status}), do: "http_#{status}"
  defp extract_reason(reason) when is_atom(reason), do: to_string(reason)
  defp extract_reason(_), do: "unknown"
end
```

Add `MyApp.Telemetry` to your supervision tree:

```elixir
# lib/my_app/application.ex
children = [
  MyApp.Telemetry,
  # ... rest of your children
]
```

<Warning>
Avoid using `url` as a Prometheus tag — URLs have unbounded cardinality and will cause memory issues. Stick to `method` and `status`.
</Warning>

## OpenTelemetry integration

Use `opentelemetry_telemetry` to bridge HTTPower telemetry events into OpenTelemetry spans:

```elixir
# mix.exs
defp deps do
  [
    {:httpower, "~> 0.9"},
    {:opentelemetry, "~> 1.0"},
    {:opentelemetry_telemetry, "~> 1.0"}
  ]
end
```

```elixir
# lib/my_app/application.ex
defmodule MyApp.Application do
  use Application

  def start(_type, _args) do
    OpentelemetryTelemetry.register_application_tracer(:httpower)

    children = [
      # ...
    ]

    Supervisor.start_link(children, strategy: :one_for_one, name: MyApp.Supervisor)
  end
end
```

HTTPower telemetry events are automatically mapped to OpenTelemetry spans with these attributes:

- HTTP requests → spans with `method`, `url`, `status` attributes
- Retries → span events
- Circuit breaker state changes → span events
- Rate limiting → span attributes

## Phoenix LiveDashboard integration

Display HTTPower metrics in Phoenix LiveDashboard:

```elixir
# mix.exs
defp deps do
  [
    {:httpower, "~> 0.9"},
    {:phoenix_live_dashboard, "~> 0.8"}
  ]
end
```

```elixir
# lib/my_app_web/telemetry.ex
defmodule MyAppWeb.Telemetry do
  use Supervisor
  import Telemetry.Metrics

  def start_link(arg) do
    Supervisor.start_link(__MODULE__, arg, name: __MODULE__)
  end

  def init(_arg) do
    children = [
      {:telemetry_poller, measurements: [], period: 10_000}
    ]

    Supervisor.init(children, strategy: :one_for_one)
  end

  def metrics do
    [
      summary("httpower.request.duration",
        unit: {:native, :millisecond},
        tags: [:method]
      ),
      counter("httpower.request.count", tags: [:method, :status]),
      counter("httpower.retry.attempts", tags: [:method]),
      counter("httpower.circuit_breaker.state_changes", tags: [:to_state]),
      counter("httpower.rate_limit.exceeded", tags: [:bucket_key])
    ]
  end
end
```

```elixir
# lib/my_app_web/router.ex
live_dashboard "/dashboard",
  metrics: MyAppWeb.Telemetry
```

## Custom logging handlers

### Correlation IDs and distributed tracing

HTTPower generates a unique `req_<16 hex chars>` correlation ID per request. In Phoenix applications, it automatically reuses the `request_id` from `Logger.metadata/0`.

To propagate correlation IDs in your telemetry handlers:

```elixir
defmodule MyApp.TracingHandler do
  def handle_event([:httpower, :request, :stop], measurements, metadata, _config) do
    correlation_id = Logger.metadata()[:httpower_correlation_id]
    duration_ms = System.convert_time_unit(measurements.duration, :native, :millisecond)

    MyApp.Tracer.record_http_call(%{
      correlation_id: correlation_id,
      method: metadata.method,
      url: metadata.url,
      status: Map.get(metadata, :status),
      duration_ms: duration_ms,
      retry_count: Map.get(metadata, :retry_count, 0)
    })
  end
end
```

## Log aggregation queries

HTTPower's built-in logger sets structured metadata on every log entry. Use these queries in your log aggregation platform:

<Tabs>
  <Tab title="Datadog">
    ```
    # Slow requests
    @httpower_duration_ms:>1000 @httpower_event:response

    # 5xx errors on POST
    @httpower_status:>=500 @httpower_method:post

    # Trace a specific request
    @httpower_correlation_id:"req_abc123"

    # Circuit breaker open events
    @httpower_event:response @httpower_status:503
    ```
  </Tab>
  <Tab title="ELK / Elasticsearch">
    ```json
    { "query": { "range": { "httpower_duration_ms": { "gt": 1000 } } } }
    ```

    ```json
    {
      "query": {
        "bool": {
          "must": [
            { "range": { "httpower_status": { "gte": 500 } } },
            { "term": { "httpower_method": "post" } }
          ]
        }
      }
    }
    ```
  </Tab>
  <Tab title="Loki / Grafana">
    ```
    # Slow requests
    {app="my_app"} | json | httpower_duration_ms > 1000

    # All 5xx errors
    {app="my_app"} | json | httpower_status >= 500

    # Trace a request
    {app="my_app"} | json | httpower_correlation_id = "req_abc123"
    ```
  </Tab>
</Tabs>

## Best practices

<CardGroup cols={2}>
  <Card title="Convert duration units" icon="clock">
    Duration measurements are in native time units. Always convert:

    ```elixir
    duration_ms = System.convert_time_unit(
      measurements.duration,
      :native,
      :millisecond
    )
    ```
  </Card>
  <Card title="Manage tag cardinality" icon="tag">
    Never use `url` as a metric tag — it has unbounded cardinality. Use `method` and `status` instead:

    ```elixir
    tags: [:method, :status]  # good
    tags: [:url]              # bad
    ```
  </Card>
  <Card title="Detach handlers in tests" icon="flask-conical">
    Always clean up telemetry handlers between tests:

    ```elixir
    setup do
      ref = make_ref()
      :telemetry.attach_many(ref, events, &handler/4, nil)
      on_exit(fn -> :telemetry.detach(ref) end)
    end
    ```
  </Card>
  <Card title="Sample high-traffic events" icon="percent">
    For very high-throughput applications, sample to control volume:

    ```elixir
    def handle_event(event, measurements, metadata, _config) do
      if :rand.uniform() < 0.1 do  # 10% sample
        process_event(event, measurements, metadata)
      end
    end
    ```
  </Card>
</CardGroup>
